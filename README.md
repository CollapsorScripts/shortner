## Roadmap
Задача: реализовать сервис для сокращения URL ссылок.

Сервис должен предоставлять API для создания коротких ссылок, предоставления статистики по короткой ссылке, выполнять редиректы и собирать статистику переходов.
Хранить данные в базе данных PostgreSQL и развертывание сервиса через **Docker**.

При составлении плана разработки ориентир был взят на простоту сервиса, возможность расширяемости сервиса, читаемость кода, возможность запуска сервиса "в один клик".

На этапе проектирования выбрал следующий стэк:
- **Fiber** - в качестве HTTP фрейморвка
> Был выбран по причине того, что проект достаточно маленький, а [Fiber](https://github.com/gofiber/fiber) минималистичен, прост и идеально подходит для быстрого старта, при этом достаточно быстрый и функциональный. Так же из коробки предоставляет контекст в рамках одного запроса ```RequestCtx```, что отлично сочетается с возможностью в handlers обращаться к БД с контекстом и дает понимание того, что контекст живет пока выполняется handler.
> В дополнение к [Fiber](https://github.com/gofiber/fiber) использовался пакет [goccy/go-json](https://github.com/goccy/go-json), так как фреймворк из коробки умеет сам преобразовывать тело запроса в ```go-структуру``` и возвращать ответ в ```json``` из ```go-структуры``` и дает возможность указать кастомный ```json-encoder/decoder```. Так как все handlers так или иначе взаимодействуют с ```json```, стандартный ```encoding/json``` намного медленнее чем [goccy/go-json](https://github.com/goccy/go-json), этим выбором я оптимизировал производительность хэндлеров, получил ускорение http-запросов и ответов.
- **PostgreSQL** - в качестве основной базы данных
> Надежная, производительная и достаточно функциональная open-source БД, отлично вписывается почти в любой проект вне зависимости от его сложности.
> Драйвер для взаимодействия с **PostgreSQL** выбрал [pgx](https://github.com/jackc/pgx) из-за простоты и функциональностью инструмента, главной причиной такого выбора стало наличие возможности из под коробки работать с контекстом, при чем в полностью автономном режиме, что прекрасно сочетается с [Fiber](https://github.com/gofiber/fiber) и его ```RequestCtx```, что дает возможность при запросах к БД прокидывать контекст из http-handler напрямую в [pgx](https://github.com/jackc/pgx), а он в свою очередь под коробкой сам определит когда завершить контекст, был ли запрос отменен до фактического исполнения запроса.
> Так же исходя из идеи проекта "простота и быстрый старт", для SQL запросов мой выбор пал на компилятор SQL - [sqlc](https://github.com/sqlc-dev/sqlc), генерирует типобезопасный код из SQL запросов, а это значит что нам не придется тратить время на рутинные задачи и сразу приступить к написанию бизнес-логики, к тому же очень хорошо подходит для простых SQL запросов, для сервиса shortener именно то что нужно.
> Так как в проект заложена идея расширяемости, нужно было предусмотреть и то, что схема БД так же может меняться, что привело бы к проблеме несоответствия схем БД, а значит нужен инструмент для управления миграциями, выбрал [golang-migrate](https://github.com/golang-migrate/migrate), имеет в арсенале буквально несколько методов из-за чего становится крайне простым в освоении, не требует дополнительно углубляться в детали реализации инструмента, при этом решает необходимую мне проблему: интегрировать миграции, откатыать их, пересоздавать БД и самое главное - версионирование миграций, что дает возможность расширять схему БД не боясь что она станет несовместимой с внесенными изменнениями.
- **Clean Env** - для управления конфигурациями.
>  [cleanenv](https://github.com/ilyakaznacheev/cleanenv) скорее мое личное предпочтение из-за удобства, да и в общем ```yaml``` конфигурации крайне наглядны, читаемы и понятны (если не ошибаюсь, то умеет работать и с .env), минималистичный, умеет сразу преобразовывать конфигурационный файлы в структуры, при этом за счет структурных тэгов расширяет функциональность и возможности, можно указать значения по умолчанию, тэги дают возможность совместить ```yaml``` конфигурацию с ```env```, что удобно в моем случае, так как системная конфигурация содержится в файле ```yaml```, а настраиваемые параметры прокидываются через контейнер посредством создания ```environment``` и в моем случае в одной структуре можно совместить параметры из всех источников.

- **Архитектура проекта**
> Приложение было разделено по слоям: handler, repository, service. Цель:
Разделить ответственность, упростить тестирование.


> То есть решил придерживаться принципа SRP, данный принцип отлично подходит для расширяемого проекта, в случае слоя http-handler будет четкое понимание зоны ответственности хэндлера, легко редактировать, легко добавлять новый функционал, четкое понимание какой файл за какие области отвечает. 


> В случае с базой данных, каждый repository предоставляет собой интерфейс, которые используют сервисы с бизнес-логикой, а значит при необходимости или желании сменить БД, достаточно будет добавить новую реализацию интерфейса и у сервисов указать какую из реализаций использовать, тем самым мы не затронем функциональность всего приложения в целом и ничего не придется подгонять под новую реализацию.

## Настройка
В папке проекта есть конфигурационные файлы:
**.env** - содержит настройки переменных окружения для контейнеров **Docker** и самих сервисов
```
DB_USER=имя пользователя БД
DB_PASSWORD=пароль БД
DB_NAME=имя базы данных
DB_PORT=порт базы данных
DB_HOST=хост базы данных (или имя контейнера)
SERVER_PORT=порт который прослушивает api gateway
```

**prod.yaml** - внутренние настройки сервера (расположение ```config/prod.yaml```)
```yaml
env: prod #окружение, для локального запуска без контейнера нужно прописать local
short_url_length: 6 #длина сокращенных ссылок
paths: #настройки фиксированных путей
  logDir: ./log #папка с логами
  logName: shortner.log #файл с логом
  migrations: #настройка миграций
    dir: "migrations" #папка с миграциями
    recreate: false #обновлять стуктуру бд новыми миграциями
    drop: false #удалять таблицы при пересоздании
server: #настройки сервера
  timeout: 30s #тайм аут сервера
```

## Установка
Скачиваем репозиторий:

```bash
git clone https://github.com/CollapsorScripts/shortner.git
```

Заходим в папку проекта:

```bash
cd shortner
```

И запускаем команду

```bash
docker compose up -d
```

Командой в одну строку:
```bash
git clone https://github.com/CollapsorScripts/shortner.git && cd shortner && docker compose up -d
```

Развернется 2 контейнера:
**service** - контейнер с api gateway
**db** - контейнер с базой данных

## Запуск
Доступно 2 варианта запуска **в контейнере Docker** и **локальный запуск**

### Запуск в контейнере
Выставляем необходимые [настройки](#настройка) и в папке проекта выполняем команду 
```bash
docker compose up -d
```

### Локальный запуск
Сервис запускается с аргументами, при локальном запуске можно указать локальную конфигурацию.

Находясь в папке проекта создаем локальную конфигурацию:
```bash
touch config/local.yaml
```
И указываем настройки локальной среды:
```yaml
env: local #локальное окружение
short_url_length: 6
database: #настройки базы данных
  host: 127.0.0.1 #локальный или глобальный IP адрес сервера с базой данных
  port: 3050 #порт который прослушивает база данных
  user: db_user #имя пользователя БД
  password: db_password #пароль БД
  name: db_name #имя БД
paths: #настройки фиксированных путей
  logDir: ./log #папка с логами
  logName: shortner.log #файл с логом
  migrations: #Настройки миграций
    dir: "internal/database/migrations" #Папка с миграциями
    recreate: false #обновлять стуктуру бд новыми миграциями
    drop: false #удалять таблицы при пересоздании
server: #настройки сервера
  port: 8010 #порт который прослушивает api gateway
  timeout: 30s #тайм аут ожидания
```

Следующим шагом собираем конфигурацию с использованием **make**, в корне проекта выполняем команду:
```bash
make
```
Или собираем вручную:
```bash
go build -o bin/shortner ./cmd/entrypoint
```

И запускаем:
```bash
./bin/shortner --config=config/local.yaml
```

## Утилиты
В проекте используется утилита **make**, в корне проекта можно обнаружить файл **Makefile**.
Техническое описание:
```bash
make - Компилирует проект в папку bin под именем shortner и запускает с аргументами --config=config/local.yaml

```

## API
### Эндпоинт генерации сокращенной ссылки
```
POST: /api/v1/shorten
```

Тело запроса:
```json
{
  "url": "URL сайта"
}
```

Тело ответа:
```json
{
  "short_url": "сокращенная ссылка (без домена)"
}
```

### Эндпоинт запроса статистики сокращенной ссылки
```
GET: /api/v1/stats/{сокращенная ссылка}
```

Тело ответа:
```json
{
	"original_url": "оригинальная ссылка",
	"clicks": "количество кликов",
	"created_at": "время создания",
	"last_accessed_at": "время последнего посещения ссылки"
}
```

### Пример
Отправляем ссылку на генерацию сокращенной ссылки:
```bash
curl -X POST http://localhost:8010/api/v1/shorten -H "Content-Type: application/json" -d '{"url":"https://example.com"}'
```

Получаем ответ:
```json
{
  "short_url": "1BjkIH"
}
```

Далее можем проверить статистику этой ссылки:
```bash
curl http://localhost:8010/api/v1/stats/1BjkIH
```

Получаем ответ:
```json
{
	"original_url": "https://example.com",
	"clicks": 7,
	"created_at": "21/04/2026 17:34",
	"last_accessed_at": "21/04/2026 21:55",
}
```

## Принятые решения
> В качесиве кода редиректа был выбран **301**, так как подразумевается что ссылка генерируется на постоянно доступный ресурс и сокращенная ссылка будет часто использоваться, **301** позволит перенести SEO и закэшировать ссылку, при повторном посещении тем же клиентом не будет необходимости выполнять повторный запрос к серверу

> Метод генерации сокращенных ссылок - генерируется случайная строка из алфавита upper case, lower case и цирф от 0 до 9
При генерации идет проверка существует ли уже такая ссылка (если при создании ссылки сервис БД отдал **nil**, значит ссылка существует и нужно сгенерировать другую). Количество символов в сокращенной ссылке управляется с помощью настройки в конфигруции ```config/prod.yaml``` параметр ```short_url_length``` , по умолчанию стоит значение 6.
